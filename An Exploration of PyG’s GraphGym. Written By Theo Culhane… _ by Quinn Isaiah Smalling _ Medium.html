Introduction

GraphGym is a powerful tool that allows users to easily design and evaluate Graph Neural Networks (GNN’s). In this tutorial, we will walk through three different use cases for GraphGym in three different domains. First, we will explore node level prediction on determining the class a document belongs to, while learning about the basics of GNN’s in GraphGym. Then, we will learn how to load your own modules into a GraphGym GNN, and how to evaluate those new modules on a link predictions collaboration dataset. Finally, we will use GraphGym to find the best GNN for predicting new molecules, using a graph level GNN pipeline.

Part 1: Node-Level Prediction and Implementing a GNN Pipeline

Motivation

Graphs are one of the most general types of data structures, allowing us to represent a wide variety of data types and the relations among those data types in a uniform and easily understandable way. However, this great flexibility comes at some cost, for one that traditional machine learning algorithms, such as multilayer perceptrons, convolutional neural networks, transformers, and other similar algorithms don’t work particularly well. This is where Graph Neural Networks come in, as a type of neural network designed specifically to make useful predictions on data formatted as a graph. While GNNs can perform very well on graph data when tuned properly and used in the right way, there are many design decisions that must go into each GNN. Further, there are many subtleties in how GNNs must be implemented that can cause bugs that manifest in complex and intricate ways. Since the difficulty and time intensity of building a robust GNN from scratch can prove daunting to beginners and experts alike, researchers have created GraphGym, a framework for rapidly building and testing different GNN models with different hyperparameters, allowing researchers and GNN users to skip the steps of reimplementing the portions of a GNN for which implementations already exist and allowing them to focus on the parts that are new, exciting, or useful for their own particular applications. In this post, we will explore some of the functionality of GraphGym, first working through a node classification dataset to understand the basics of how to use GraphGym to setup and execute a GNN pipeline, then going through a link prediction example to understand how to customize portions of the GNN created, and finally work through a graph level prediction example to understand how to use GraphGym to find the best GNN design for an application.

Data

For the first example we work through, we will be looking at how to do node classification on the PubMed dataset proposed in “Revisiting Semi-Supervised Learning with Graph Embeddings”. Each dataset contains a bag of words representation of documents as the node features, and the links represent citation links between those documents in an undirected graph structure, meaning that if document i cites document j, there will be a 1 in the adjacency matrix at both location ij and ji. We then use a GNN to predict the class of each document. This dataset is built into PyG, making loading and working with the dataset easier than loading in a custom dataset, and so it is a good candidate for this post, in which the focus is on the GNN pipeline as a whole, and not just loading data.

Defining Hyperparameters

The first step in getting your GNN pipeline running in GraphGym is to create a YAML file containing information about your dataset, GNN architecture, and hyperparameters for your initial experiments. For our PubMed dataset example, we will use a YAML file that looks like this, saved at configs/node.yaml:

out_dir: results
dataset:
  format: PyG
  name: PubMed
  task: node
  task_type: classification
train:
  batch_size: 128
  eval_period: 1
  ckpt_period: 100
  sampler: full_batch
model:
  type: gnn
  loss_fun: cross_entropy
gnn:
  layers_pre_mp: 1
  layers_mp: 3
  layers_post_mp: 1
  dim_inner: 128
  layer_type: sageconv
  stage_type: skipsum
  batchnorm: True
  act: prelu
  dropout: 0.1
  agg: mean
  normalize_adj: False
optim:
  optimizer: adam
  base_lr: 0.01
  max_epoch: 200

Walking through this YAML, we first see the definition of the directory that we want to save all our results for our experiments into under the key out_dir. Next, we define some information about the dataset that we want to run our experiments on. In this case, we know that we want to use the PubMed dataset, which is formatted as a PyG (pytorch-geometric) dataset. Currently, GraphGym supports PyG and OGB formatted datasets, with dataset loaders pre-implemented for a variety of different PyG and OGB datasets. Next, we see that for our example, we want to perform node classification, as we have defined our dataset task to be node and our task_type to be classification.

Next, we define the hyperparameters we want to use for training in the train section of our YAML file. Specifically, in this case, we decide that we want to use a batch size of 128 and generate statistics on our model’s performance on the dev and test sets every epoch, along with checkpointing the model every 100 epochs. Finally, we will use the full batch for each epoch of training, instead of subsampling only part of our data in each epoch.

Then, we begin to define how our model will be built using the model and gnn sections of our YAML file. Note that while we define many hyperparameters in our specific example, in practice you don’t need to define any hyperparameters you don’t want to. For any hyperparameters that you leave unspecified, GraphGym will fill in default values. For more information about what default values GraphGym uses, see config.py in the GraphGym repository.

For our example, we will use a GNN that is evaluated using cross entropy (a standard loss function for classification tasks), as defined in the model section of the YAML file. Then each GNN layer, we will use 1 neural network layer before message passing, three layers of message passing, and then one neural network layer after message passing, and a hidden dimension inside our layers of 128. We will use the SageConv variety of GNN conv layer, using a Skip-Sum residual layer for residual connections. Next, we define that we want to use batch-normalization in our GNN, we want PReLU as our non-linear activation function, that we want a dropout rate of 0.1, and that we want to use mean aggregation for our aggregation step in our GNN. Finally, we define that we do not want to normalize the adjacency matrix for our graph.

Finally, in our YAML file, we define some hyperparameters about how our GNN will be optimized. Specifically, we define that we want to use the ADAM optimizer with a learning rate of 0.01, and we will run optimization for a maximum of 200 epochs.

Running Experiments

We will explore the different ways to run experiments in more depth later in this post, but for now, to test that we’ve run all the steps properly up to this point, we can run the following command.

!python main_pyg.py --cfg configs/node.yaml --repeat 3

If all has been done correctly, you should see GraphGym train a model 3 different times with different random initializations for 200 epochs and save the results of those three runs into results/node. You should see separate logging information for each of the three runs stored separately, along with combined statistics about how the model performed across all three runs aggregated into a folder called agg.

We can also run a full set of experiments (also explored in more depth later), by following this series of steps:

First, create a “grid” file to define how you will perturb your model hyperparameters during your set of experiments. For our example, we use the following grid file, saved at grids/node_grid.txt:

model.graph_pooling gp ['add','mean','max']
gnn.layers_pre_mp l_pre [1,2]
gnn.layers_mp l_mp [2,4,6]
gnn.layers_post_mp l_post [1,2]
gnn.stage_type stage ['stack','skipsum','skipconcat']
gnn.dim_inner dim [64,128,256]
optim.base_lr lr [0.001,0.01]
optim.max_epoch epoch [100,200,300]

In this grid file, we define which hyperparameters we want to perturb, along with the possible values we want those hyperparameters to be able to take on.

Then, we run:

!python configs_gen.py --config configs/node.yaml --grid grids/node_grid.txt --out_dir configs --sample --sample_num 20

This will generate a set of config files that correspond to the different ways that we perturbed the hyperparameters. If we don’t specify — sample, it will create a config file for every combination of the different hyperparameters specified in the grid file. However, since we do specify sample, we will only generate 20 config files, which will be a random subsample of the total set of possible hyperparameter configurations.

Next, run:

!bash parallel.sh configs/node_grid_node_grid 3 1 main_pyg

This will run all the config files created by the previous command. In this case, because we specified 1 for the third parameter, the config files will be run in sequence, but if that number is increased you can run multiple experiments in parallel.

Finally, run:

!python agg_batch.py --dir results/node_grid_node_grid

This will aggregate all the results from the various hyperparameter configurations into a single set of CSVs, so that further analysis can be done on which hyperparameter configuration is best. Again, this will be explored more later in greater detail, including examples of how to visualize these CSVs and make good decisions about the configurations.

Part 2: Edge-Level Prediction and Customizing Modules

Now, we will take some of the lessons we learned above and expand on them, learning to customize modules to improve link prediction. We now know some of the many examples of data that easily can be represented as graphs. Link prediction builds on this representation, and is commonly used for a variety of industry tasks such as:

  • Friend prediction in a social network, with edges representing possible friendships between users. Link prediction in this context can simplify finding a more authentic group on social media.
  • Prediction of negative drug interactions in a network of drugs. With a large set of pharmaceutical drugs being used to cure a variety of ailments, understanding the interactions between them has become even more important. With link prediction, we can quickly explore this large space, without relying on costly physical experiments.
  • Recommending items to users in a bipartite graph of users and items, which can increase satisfaction among users.

While there are many examples of possible domains that edge level prediction can be used on, all tasks utilize a similar underlying methodology. We will now use PyG’s GraphGym to explore that methodology, specifically focusing on how to modify sections to achieve better results, with less time spent focusing on implementing each specific module.

To aid with that goal, we will continue in our Colab. The second section provides all code necessary to step through edge level prediction, as well as how to customize modules. We will show how to modify the basic setup, what GraphGym is doing for us, and how to evaluate these modifications.

First, we need a dataset. We will be using Open Graph Benchmark’s (OGB) link prediction datasets, specifically the collaborations network dataset. This undirected dataset represents authors as nodes and collaborations as edges, while also including a 128-dimension feature vector embedding the words in papers they published. Additionally, the edges have a feature corresponding to the number of collaborations between the two authors in that given year, as well as the year the collaborations took place. The data is segmented by time, with collaborations before 2017 making up the training data, 2018 being testing data, and 2019 representing validation data. Time sequence segmentation mimics how the GNN would be used in the real world, to predict upcoming collaborations. This graph is categorized as a dynamic multigraph, since there can be multiple edges between different nodes, as authors can collaborate in multiple years.

OGB also provides additional datasets that can be loaded in, including a drug-drug interaction graph, a protein association graph, and a knowledge graph from the WikiData knowledge base. While loading in custom datasets is possible, we will be focusing on GraphGym’s custom modules. Resources for loading in additional graphs can be found here, and the next steps are all identical regardless of the dataset being used.

Modifying the GNN in GraphGym

Now that we understand how to set up some GraphGym basics, we can explore how to modify our GNN. First, we will change some of the predefined GraphGym parameters. Then, we will define our own parameters, for a customizable GNN, and run an experiment to test our changes.

Using predefined GraphGym parameters is simple. While nearly all of the parameters are modifiable, not all variants are currently accessible. In the GNN section of the example yaml file, we simply update the parameter to be the new parameter we want. We previously looked at how to modify batch size, inner dimensions, and epochs. The principle is the same- we simply update the agg: parameter to be the new aggregation function we want. For example, if we want to use add instead of mean for our aggregation function, we would simply change agg: mean to agg: add. The simple, modular, and intuitive design of GraphGym is what makes it such a powerful tool. These predefined settings can be explored in graphgym/contrib, which is constantly being updated by GraphGym users to add additional functionality.

Now, we will look at defining our own modular parameters. To better understand this, we will compare three different activation functions: leaky ReLU with a negative slope of 0.3, which has already been defined within GraphGym, the swish activation function, and leaky ReLU with a negative slope of 0.5. To find the definition of leaky ReLU, go to GraphGym/graphgym/contrib/act/example.py.

import torch
import torch.nn as nn

from graphgym.config import cfg
from graphgym.register import register_act


class SWISH(nn.Module):
    def __init__(self, inplace=False):
        super().__init__()
        self.inplace = inplace

    def forward(self, x):
        if self.inplace:
            x.mul_(torch.sigmoid(x))
            return x
        else:
            return x * torch.sigmoid(x)


register_act('swish', SWISH(inplace=cfg.mem.inplace))

register_act('lrelu_03',
             nn.LeakyReLU(negative_slope=0.3, inplace=cfg.mem.inplace))

register_act('lrelu_05',
             nn.LeakyReLU(negative_slope=0.5, inplace=cfg.mem.inplace))

Here, we see two activation functions that have been registered: swish and lrelu_3. For simplicity, we will define a simple activation function and then use graphgym to compare these two functions.

First, simply define your own activation function as above. Ours is the “lrelu_05”, at the bottom of this code block.

To get a better idea of how to perturb our model, we will also explore modifying or swapping layers in a GNN. To understand this, simply open the GraphGym/graphgym/contrib/layer directory. In the example.py file, we have an example of how to define a new Convolution layer, ‘exampleconv’. Essentially, we can define a new layer, then register that layer to use it in our GNN. An important note is that we need to make sure that we define our layer with our use case in mind- it doesn’t make sense to use this example layer for link prediction since we are using edge embeddings and our layer doesn’t contain any.

import torch
import torch.nn as nn
from torch.nn import Parameter
from torch_geometric.nn.conv import MessagePassing
from torch_geometric.nn.inits import glorot, zeros

from graphgym.config import cfg
from graphgym.register import register_layer

class ExampleConvLayer(MessagePassing):
    r"""Example GNN layer

    """
    def __init__(self, in_channels, out_channels, bias=True, **kwargs):
        super(ExampleConvLayer, self).__init__(aggr=cfg.gnn.agg, **kwargs)
        self.in_channels = in_channels
        self.out_channels = out_channels
        self.weight = Parameter(torch.Tensor(in_channels, out_channels))
        if bias:
            self.bias = Parameter(torch.Tensor(out_channels))
        else:
            self.register_parameter('bias', None)
        self.reset_parameters()

    def reset_parameters(self):
        glorot(self.weight)
        zeros(self.bias)

    def forward(self, x, edge_index):
        """"""
        x = torch.matmul(x, self.weight)

        return self.propagate(edge_index, x=x)

    def message(self, x_j):
        return x_j

    def update(self, aggr_out):
        if self.bias is not None:
            aggr_out = aggr_out + self.bias
        return aggr_out

    def __repr__(self):
        return '{}({}, {})'.format(self.__class__.__name__, self.in_channels,
                                   self.out_channels)


class ExampleConv(nn.Module):
    def __init__(self, dim_in, dim_out, bias=False, **kwargs):
        super(ExampleConv, self).__init__()
        self.model = ExampleConvLayer(dim_in, dim_out, bias=bias)

    def forward(self, batch):
        batch.node_feature = self.model(batch.node_feature, batch.edge_index)
        return batch


# Remember to register your layer!
register_layer('exampleconv', ExampleConv)

Part 3: Graph-Level Prediction and Finding the Best GNN Design

Finally, let’s discuss how GraphGym can help us find the best GNN design for a given problem.

To motivate this, we will use the “MoleculeNet HIV” dataset, which consists of 41,127 graphs each representing a molecule (where nodes are atoms and edges are chemical bonds between atoms). Node input features are 9-dimensional, containing atomic information such as the atomic number and formal charge, among others. The ultimate graph-level prediction task is to classify whether or not each molecule inhibits HIV. Because this dataset is so skewed (only 1.4% positive labels), ROC AUC is typically used as an evaluation metric instead of accuracy.

gnn.layers_mp l_mp [4,6]
gnn.layers_post_mp l_post [2,3]
gnn.stage_type stage ['skipconcat']
optim.base_lr lr [0.005,0.01]

Note that each parameter specified in this file will replace a parameter in the example_graph.yaml file! For example, by specifying this grid, I am changing the stage type from ‘stack’ to ‘skipconcat’.

*If you decide to create your own .txt file to specify your grid search space, rather than modify the example.txt file, make sure to make the necessary changes to run_batch_pyg.sh so that the program knows where to find your file.

Now, we can run our experiments with the following line:

!bash run_batch_pyg.sh #run our specified experiments

This will take some time, depending on our total number of experiments. Ultimately, the results of each experiment will be saved in the results/example_grid_graph_sample directory, and even more conveniently, the results will also be aggregated and saved in the results/example_grid_graph_sample/agg directory. Here, we can find the csv files like val_best.csv, which saves the model from the epoch with the best validation performance for each of our different experiments.

Visualization

In many cases, though, we’d ideally like to visualize the results from our experiments so that we can compare the different models. Outside of the Run directory in GraphGym/analysis, we can find a file titled example.ipynb that contains all of the relevant code to visualize our results! First, we can copy over the first cell from that file into our notebook and run it to define the relevant visualization functions (it’s quite long, so not included here, but you can simply copy and paste). Then, we can run the following line in our Colab to get our results:

# analysis for all dataset in the batch
experiment_name = 'example_graph_grid_example'
plot_analysis(experiment_name, division='val', dataset="all", metric='auc', rank_resolution=0)

Here, we can compare each of the specified values for our experiments with their isolated AUC distributions, rank averages, and rank distributions. Note that these are for validation performance. Obviously, a higher AUC is better, but we actually want a lower rank. For example, we have two values for our number of MP layers, 4 and 6. We can rank these values based on their performance in each experiment, either first or second. Thus, rank average is computing just that: each specified value’s average ranking compared to the others for each experiment.
